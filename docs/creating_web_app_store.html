Building Open Web Directories and Stores

Help Users and Developers Connect

The Mozilla Labs Open Web Application platform provides features in the web browser that allow web sites to be recognized as "web applications" - rich, interactive experiences, represented by icons, which can be installed and synchronized across devices, and distributed through directories and stores.

Integration with the Browser

The application platform provides the navigator.apps JavaScript object, which your store will use to communicate with the browser.

A web application is identified by its domain; each domain may contain exactly one application. As a directory or store, you may use the navigator.apps API to prompt the browser to install an application by invoking the install method, providing the URL of the application's manifest and whatever optional installation metadata you require. This metadata can be accessed by application developers to check back with you for purchase confirmation, single sign-on, or other services.

Using getInstalledBy

The navigator.apps.getInstalledBy method allows you to get a list of the applications installed in the current user agent by your domain. Note that you will not see applications installed by other domains - only the ones that you put there.

You can use this method to determine whether the user's installed applications match your expectations, or to implement a "resync" feature if the user signs into your site from a new profile.

Hosting or Not?

There is no requirement for a store or directory to host the applications that it describes. The store can simply invoke install with the site's manifest URL.

For applications which do not have a server logic component (that is, which are implemented entirely in client-side JavaScript, HTML, and CSS), you may want to implement a hosting feature. To do this, you will need to assign a different domain name to each application and implement server logic to serve the application manifest for each of them.

If you are hosting the application logic yourself, you can easily maintain a session with the user to track preferences, proof-of-purchase, or other additional services. If you are providing services to a remote website, you will need to do some additional work to support a distributed authentication system.
